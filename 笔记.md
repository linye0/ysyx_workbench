# Makefile笔记
.PHONY标识的是一个伪目标
```makefile
.PHONY: clean
clean:
  rm -f m.txt
  rm -f x.txt
```
这样Makefile就不会把clean视为一个文件名，但一般来说，并不需要用.PHONY标识clean等约定俗成的伪目标名称，除非有人故意搞破坏，手动创建名字叫clean的文件。

Makefile会为每一条指令创建一个单独的Shell环境，类似cd ..这样的指令不会影响目录，如果要实现目录切换，应该把多条指令并成一条：

```makefile
cd_ok:
	pwd; cd ..; pwd;
```

或者用&&连接，这样前面的指令若出错后面的就不会执行：

```makefile
cd_ok:
  cd .. && pwd
```

对于执行可能出错，但不影响逻辑的命令，可以用-忽略。

```makefile
ignore_error:
	-rm zzz.txt
	echo 'ok
```

使用变量以避免重复文件名：
```makefile
# $(wildcard *.c) 列出当前目录下的所有 .c 文件: hello.c main.c
# 用函数 patsubst 进行模式替换得到: hello.o main.o
OBJS = $(patsubst %.c,%.o,$(wildcard *.c))
TARGET = world.out

$(TARGET): $(OBJS)
	cc -o $(TARGET) $(OBJS)

clean:
	rm -f *.o $(TARGET)
```

自动变量：
$@代表目标文件，$^代表所有依赖文件

模式规则：
```makefile
OBJS = $(patsubst %.c,%.o,$(wildcard *.c))
TARGET = world.out

$(TARGET): $(OBJS)
	cc -o $(TARGET) $(OBJS)

# 模式匹配规则：当make需要目标 xyz.o 时，自动生成一条 xyz.o: xyz.c 规则:
%.o: %.c
	@echo 'compiling $<...'
	cc -c -o $@ $<

clean:
	rm -f *.o $(TARGET)
```

自动生成依赖，可以用于解决前面所编写的Makefile无法识别.c文件依赖哪个.h文件的问题，采用的方法是用几个.d文件把依赖关系存下来，然后再include到Makefile里面：

```makefile
# 列出所有 .c 文件:
SRCS = $(wildcard *.c)

# 根据SRCS生成 .o 文件列表:
OBJS = $(SRCS:.c=.o)

# 根据SRCS生成 .d 文件列表:
DEPS = $(SRCS:.c=.d)

TARGET = world.out

# 默认目标:
$(TARGET): $(OBJS)
	$(CC) -o $@ $^

# xyz.d 的规则由 xyz.c 生成:
%.d: %.c
	rm -f $@; \
	$(CC) -MM $< >$@.tmp; \
  # cc -MM main.c会生成main.o: main.c main.h，这样可以把main.c对main.h的依赖加进来，先把这条规则储存在main.d.tmp文件当中
	sed 's,\($*\)\.o[ :]*,\1.o $@ : ,g' < $@.tmp > $@; \
  # sed指令是对main.d.tmp当中的内容进行处理，并且输出到main.d当中，其中模式替换的部分，是把main.d.tmp当中的main.o换成main.o main.d，也就是说main.d最终的内容是main.o main.d: main.c main.h，这样可以让main.d也依赖main.h，之后可以自动更新
	rm -f $@.tmp

# 模式规则:
%.o: %.c
	$(CC) -c -o $@ $<

clean:
	rm -rf *.o *.d $(TARGET)

# 引入所有 .d 文件，这样就把之前.d当中编写的规则引入该Makefile文件了
include $(DEPS)
```

在建立build文件夹专门存放Makefile生成文件之后，Makefile最终如下：

```makefile
SRC_DIR = ./src
BUILD_DIR = ./build
TARGET = $(BUILD_DIR)/world.out

CC = cc
CFLAGS = -Wall

# ./src/*.c
SRCS = $(shell find $(SRC_DIR) -name '*.c')
# ./src/*.c => ./build/*.o
OBJS = $(patsubst $(SRC_DIR)/%.c,$(BUILD_DIR)/%.o,$(SRCS))
# ./src/*.c => ./build/*.d
DEPS = $(patsubst $(SRC_DIR)/%.c,$(BUILD_DIR)/%.d,$(SRCS))

# 默认目标:
all: $(TARGET)

# build/xyz.d 的规则由 src/xyz.c 生成:
$(BUILD_DIR)/%.d: $(SRC_DIR)/%.c
	@mkdir -p $(dir $@); \
	rm -f $@; \
	$(CC) -MM $< >$@.tmp; \
	sed 's,\($*\)\.o[ :]*,$(BUILD_DIR)/\1.o $@ : ,g' < $@.tmp > $@; \
	rm -f $@.tmp

# build/xyz.o 的规则由 src/xyz.c 生成:
$(BUILD_DIR)/%.o: $(SRC_DIR)/%.c
	@mkdir -p $(dir $@)
	$(CC) $(CFLAGS) -c -o $@ $<

# 链接:
$(TARGET): $(OBJS)
	@echo "buiding $@..."
	@mkdir -p $(dir $@)
	$(CC) -o $(TARGET) $(OBJS)

# 清理 build 目录:
clean:
	@echo "clean..."
	rm -rf $(BUILD_DIR)

# 引入所有 .d 文件:
-include $(DEPS)
```

Makefile里的变量：
1. 递归调用拓展型变量(=)
```makefile
foo = $(bar)
bar = $(ugh)
ugh = Huh?
```

2. 简单扩展型变量(:=)
```makefile
x := foo
y := $(x) bar
x := later
#等同于
y := foo bar
x := later
```

3. 条件变量赋值操作符(?=)
```makefile
FOO ?= bar
#和下面的语句严格等同
ifeq ($(origin FOO), undefined)
  FOO = bar
endif
```

4. 追加文本
```makefile
objects += another.o
#等同于
objects := $(objects) another.o
```

mepc
Event是定义在am层的结构体，不与架构相绑定：
```c
typedef struct Event {
  enum { ... } event;
  uintptr_t cause, ref;
  const char *msg;
} Event;
```
其中event表示事件编号, cause和ref是一些描述事件的补充信息, msg是事件信息字符串, 我们在PA中只会用到event. 然后, 我们只要定义一些统一的事件编号(上述枚举常量), 让每个架构在实现各自的CTE API时, 都统一通过上述结构体来描述执行流切换的原因, 就可以实现切换原因的抽象了.

CTE也是定义在am层的结构体，但是会和架构绑定，包含的信息包括寄存器状态等，还有两个统一的api：

bool cte_init(Context* (*handler)(Event ev, Context *ctx))用于进行CTE相关的初始化操作. 其中它还接受一个来自操作系统的事件处理回调函数的指针, 当发生事件时, CTE将会把事件和相关的上下文作为参数, 来调用这个回调函数, 交由操作系统进行后续处理.
void yield()用于进行自陷操作, 会触发一个编号为EVENT_YIELD事件. 不同的ISA会使用不同的自陷指令来触发自陷操作, 具体实现请RTFSC.

```c
bool cte_init(Context*(*handler)(Event, Context*)) {
  // initialize exception entry
  asm volatile("csrw mtvec, %0" : : "r"(__am_asm_trap));

  // register event handler
  user_handler = handler;

  return true;
}
```

异常处理逻辑：
先把__am_asm_trap()函数的入口地址存入mtvec，这个函数可以视为正式异常处理前的预处理过程，内容为：
```asm
__am_asm_trap:
  addi sp, sp, -CONTEXT_SIZE

  MAP(REGS, PUSH)

  csrr t0, mcause
  csrr t1, mstatus
  csrr t2, mepc

  STORE t0, OFFSET_CAUSE(sp)
  STORE t1, OFFSET_STATUS(sp)
  STORE t2, OFFSET_EPC(sp)

  # set mstatus.MPRV to pass difftest
  li a0, (1 << 17)
  or t1, t1, a0
  csrw mstatus, t1

  mv a0, sp
  call __am_irq_handle
  mv sp, a0

  LOAD t1, OFFSET_STATUS(sp)
  LOAD t2, OFFSET_EPC(sp)
  csrw mstatus, t1
  csrw mepc, t2

  MAP(REGS, POP)

  addi sp, sp, CONTEXT_SIZE
  mret
```
负责把Context压入栈中作为参数，然后调用__am_irq_handle():
```c
Context* __am_irq_handle(Context *c) {
  if (user_handler) {
    Event ev = {0};
    switch (c->mcause) {
      case 0x8ul:
      case 0x9ul:
      case 0xbul:
      {
        c->mepc = c->mepc + 4;
        if (c->GPR1 == -1) {
          ev.event = EVENT_YIELD;
        } else {
          ev.event = EVENT_SYSCALL;
        }
      }
      break;
  #if defined(CONFIG_ISA64)
      case 0x8000000000000007:
  #endif
      case 0x80000007:
      {
        ev.event = EVENT_IRQ_TIMER;
      }
      break;
    default: ev.event = EVENT_ERROR; break;
    }
    size_t mscratch = 0, sp;
    asm volatile("csrr %0, mscratch\nmv %1, sp" : "=r"(mscratch), "=r"(sp));
    // printf("mscratch = %d, sp = %d\n", mscratch, sp);
     // printf("- c: %p, c->np: %d, cp->gpr[sp]: %p, mscratch = %d, sp = %p\n",
           // c, c->np, c->gpr[2], mscratch, sp);
    c = user_handler(ev, c);
    asm volatile("csrr %0, mscratch\nmv %1, sp" : "=r"(mscratch), "=r"(sp));
     // printf("= c: %p, c->np: %d, cp->gpr[sp]: %p, mscratch = %d, sp = %p\n",
       //    c, c->np, c->gpr[2], mscratch, sp);
    assert(c != NULL);
  }

  return c;
}
```
会根据context信息生成event信息，然后调用user_handler()函数，进行正式处理.	

异常处理部分的调用链：
(am层)yield->(nemu层)isa_raise_intr->pc指向mtvec(__am_asm_trap)->(am层)__am_irq_handle

根据异常处理的逻辑，我们也可以思考出进程操作的实现：

![](./imgs/2025-08-08-20-00-31.png)

只需要在__am_asm_trap调用__am_irq_handle的过程当中进行一下sp的切换（切换到另一个进程的sp），然后通过__am_asm_trap的下半部分就可以自动把当前的context指向这个进程的context，也就是说完成了进程的切换.而这个过程可以通过用__am_irq_handle返回目标进程的Context*指针实现.

由于有很多的进程可能会同时存在，我们用PCB来对这些信息进行维护，除了Context，还有栈空间也是和进程相关的，所以PCB当中的成员如下：
```c
typedef union {
  uint8_t stack[STACK_SIZE];
  struct {
    Context *cp;
  };
} PCB;
```

# yield-os相关梳理：进程切换的方法

异常处理相关代码的执行流程回顾：
首先，调用cte_init:
```cpp
bool cte_init(Context*(*handler)(Event, Context*)) {
  // initialize exception entry
  asm volatile("csrw mtvec, %0" : : "r"(__am_asm_trap));

  #ifdef CONFIG_ISA64
  asm volatile("csrw mstatus, %0" : : "r"(0xa00001808));
#else // __risv32
  // M-mode
  asm volatile("csrw mstatus, %0" : : "r"(0x1800));
#endif
  // register event handler
  user_handler = handler;

  return true;
}
```
这段代码会把一个默认的__am_asm_trap函数入口值赋给mtvec，并且将user_handler的值设为handler, handler的具体内容先不看，__am_asm_trap在nemu项目当中的内容如下：
```asm

  MAP(REGS, PUSH)

  csrr t0, mcause
  csrr t1, mstatus
  csrr t2, mepc

  STORE t1, OFFSET_STATUS(sp)
  STORE t2, OFFSET_EPC(sp)

  # set mstatus.MPRV to pass difftest (don't need now)
  # li a0, (1 << 17)
  # or t1, t1, a0
  # csrw mstatus, t1

  mv a0, sp
  call __am_irq_handle
  mv sp, a0

  LOAD t1, OFFSET_STATUS(sp)
  LOAD t2, OFFSET_EPC(sp)
  csrw mstatus, t1
  csrw mepc, t2

  MAP(REGS, POP)

  addi sp, sp, CONTEXT_SIZE
  mret
```

程序在执行ecall指令的时候会将pc的值置为mtvec的值，并且把mepc设为当前的pc值，pc在mtvec所在地方继续执行，就会进入__am_asm_trap所对应的代码段，其行为是:先把当前的context内容存在栈上，包括regs和csr的状态，然后使用a0参数（指向栈上的这一段context）调用__am_irq_handler函数，函数执行完毕之后，从栈上回复context的内容，然后执行mret，返回异常处理之前执行的代码.

__am_irq_handle的代码如下:

```cpp
Context* __am_irq_handle(Context *c) {
  if (user_handler) {
    Event ev = {0};
    switch (c->mcause) {
      case 0x8ul:
      case 0x9ul:
      case 0xbul:
      {
        c->mepc = c->mepc + 4;
        if (c->GPR1 == -1) {
          ev.event = EVENT_YIELD;
        } else {
          ev.event = EVENT_SYSCALL;
        }
      }
      break;
  #if defined(CONFIG_ISA64)
      case 0x8000000000000007:
  #endif
      case 0x80000007:
      {
        ev.event = EVENT_IRQ_TIMER;
      }
      break;
    default: ev.event = EVENT_ERROR; break;
    }
    size_t mscratch = 0, sp;
    asm volatile("csrr %0, mscratch\nmv %1, sp" : "=r"(mscratch), "=r"(sp));
    // printf("mscratch = %d, sp = %d\n", mscratch, sp);
     // printf("- c: %p, c->np: %d, cp->gpr[sp]: %p, mscratch = %d, sp = %p\n",
           // c, c->np, c->gpr[2], mscratch, sp);
    c = user_handler(ev, c);
    asm volatile("csrr %0, mscratch\nmv %1, sp" : "=r"(mscratch), "=r"(sp));
     // printf("= c: %p, c->np: %d, cp->gpr[sp]: %p, mscratch = %d, sp = %p\n",
       //    c, c->np, c->gpr[2], mscratch, sp);
    assert(c != NULL);
  }

  return c;
}
```

这段函数会根据传入的context相关值来初始化一个Event变量，其中包含了user_handler函数执行所需要的信息，然后调用user_handler进行具体异常行为的处理，最后返回context，最后返回context的指针(注意，所有的函数都在用指针传参，所以这个指针的值可能是会被改变的)。

所以进程切换的思想就很简单了，我们只需要在user_handler把context所指向的值进行改变，这样的话__am_asm_trap当中的这段代码

```asm
mv a0, sp
call __am_irq_handle
mv sp, a0
```

执行完之后，sp的值就指向了一段新的context栈空间，紧接着，就会用这个栈空间的内容来对context进行恢复，如此实现context的切换。

我们可以用PCB结构去把多个context所对应的栈空间储存起来：
typedef union {
  uint8_t stack[STACK_SIZE];
  struct {
    Context *cp;
  };
} PCB;

在yield-os当中，我们希望程序进行进程切换之后能够自动执行一段函数，所以要用到kcontext，kcontext就是一段预定义好的context，这段预定义好的context当中，mepc值被设为entry的入口地址，所以根据这段context进行context的复原（出栈），再执行mret之后，pc值就会被设为entry的入口地址，如此实现函数的执行。

![](./imgs/2025-10-29-17-23-10.png)

kcontext的代码如下：
```cpp
Context *kcontext(Area kstack, void (*entry)(void *), void *arg) {
  Context *p = (Context *)(kstack.end - sizeof(Context));

  p->mepc = (uintptr_t)entry;
  p->gpr[r_a0] = (uintptr_t)arg;

#ifdef CONFIG_ISA64
  p->mstatus = 0xa00001808;
#else // __risv32
  p->mstatus = 0x1800;
#endif
  p->np = PRV_M;
  return p;
}
```

需要注意的是，PA当中有这么一段话，"此外, kcontext()要求内核线程不能从entry返回, 否则其行为是未定义的"，这是因为kcontext的entry函数不是通过常规调用进行的，而是通过直接跳转的指令，也就是说，在调用entry函数之前，并没有执行过寄存器压栈的操作，所以如果kcontext当中有ret这样的指令，就会导致pc被指向栈上一个无效的值，导致程序出现错误，那么，kcontext函数要怎么结束呢？答案是通过yield函数，请求os介入，如yield-os当中的kcontext的entry函数，是这样写的:
```cpp
static void f(void *arg) {
  while (1) {
    putch("?AB"[(uintptr_t)arg > 2 ? 0 : (uintptr_t)arg]);
    for (int volatile i = 0; i < 100000; i++) ;
    yield();
  }
}
```

在执行玩yield之后，会执行user_handler，这边的user_handler是一个调度函数，用于在两个f()，其实就是两个线程当中进行切换:

```cpp
static Context *schedule(Event ev, Context *prev) {
  current->cp = prev;
  current = (current == &pcb[0] ? &pcb[1] : &pcb[0]);
  return current->cp;
}
```

这样实现两个进程之间的切换。
